## 의존성과 순서 평가

- 권장 구현 순서 `WP-001` → `WP-002` → `WP-003` → `WP-004` → `WP-005` → `WP-006`은 계약, 입력·데이터, 분석, 포트폴리오, 실행 증거, 웹 통합의 선행 관계와 일치한다.
- `WP-005`의 저장·상태 핵심은 `WP-002` 이후 병렬 개발할 수 있지만 종단 간 완료 증거는 `WP-003`, `WP-004` 이후에 닫는다고 구분되어 있어 조기 완료 오인을 방지한다.
- 각 패킷은 목표, 기준선 추적, 허용 경로·컴포넌트, 금지 범위, 의존성, 구현 개요, 테스트·수용 증거, 호환성 영향, 롤백 경계와 완료 상태를 포함한다.
- 모든 코드 패킷은 실제 경로·스택 결정 전 차단되며, 도메인 정책이 필요한 패킷도 대응 `OPEN-IMPL-*` 결정 전에는 승인·실행할 수 없다. 독립 승인 시 필요한 선행 조건이 식별되어 있다.

## 검증, 마이그레이션, 롤아웃 및 롤백 평가

- `TEST-001`~`TEST-012`는 계약·상태 전이, 입력 경계, 계산 대조, 미래 데이터 누수, 적격성, 예측·백테스트, 정책 충돌, 부분 저장·멱등성, 상태 가시성, 웹 흐름, 접근성·반응형, 종단 간 재현을 포함한다.
- 각 검증은 기준선 ID와 기대 증거를 명시하며, 미결정 값이 필요한 시험은 임의 기본값으로 통과시키지 않고 `BLOCKED_BY_OPEN_DECISION`으로 보고하도록 제한한다.
- `MIG-001`~`MIG-004`는 최초 계약, 공급자 격리, 모델·정책 버전, 실행 증거 형식의 호환성 경계를 다룬다. 기존 제품 데이터와 소비자가 없다는 현재 조건도 명시한다.
- `ROLL-001`~`ROLL-004`는 단계별 진입 조건과 실패 시 되돌릴 모듈·개발 데이터 범위를 구분한다. 자동 외부 배포와 실제 사용자 데이터 전환은 금지되어 현재 개발 범위에 적합하다.
- 실행 증거의 부분 실패, 완료 조건, 멱등·충돌 및 재실행은 `WP-005`와 `TEST-008`에 포함되어 승인된 아키텍처의 복구 계약을 검증할 수 있다.

## 사람의 미결정 사항

- `OPEN-IMPL-001`~`OPEN-IMPL-009`: 시장·종목 규칙, 예측 기간, 데이터 출처, 대형주 정책, 포트폴리오 제약, 후보 범위, 반올림·허용오차, 최신성, 지원 화면 크기.
- `OPEN-IMPL-010`: 언어, 웹 프레임워크, 테스트 러너와 실제 프로젝트 경로. 모든 코드 패킷 승인 전에 프로젝트 책임자와 개발자가 결정해야 한다.
- `OPEN-IMPL-011`: 개발 실행 증거 저장 기술, 최소 보존 범위와 정리 절차. `WP-005` 승인 전에 개발자와 프로젝트 책임자가 결정해야 한다.

이 항목들은 구현 계획 승인 자체를 막는 누락이 아니라 각 작업 패킷의 승인·실행 전에 해소하도록 명시된 진입 조건이다. 해당 결정을 하지 않은 채 패킷을 승인하거나 검증을 합격 처리해서는 안 된다.
